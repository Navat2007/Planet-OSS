# PLANET OSS — Пошаговый план разработки (демо RTS в стиле Generals: Zero Hour)

> Цель: сетевая RTS 1v1 / 2v2 / 3v3 / 4v4, против ИИ или игрока. Референс — Generals: Zero Hour.
> Эталонный геймплей: см. прикреплённое видео (полевой бой двух фракций на полигоне).
> Схема управления: см. `Docs/Controls.md` (учитывается в Фазах 2.1, 4, 5, 7).
> Документ — живой, движемся по фазам сверху вниз, каждая фаза имеет критерии готовности и тесты.

## Стек проекта (на текущий момент)

- Unity + URP 17.5
- Input System 1.19 (новый ввод)
- AI Navigation 2.0 (NavMesh) — для движения юнитов
- Test Framework 1.7 — для автотестов (EditMode + PlayMode)

## Ключевое архитектурное решение: сетевая модель

Выбрана модель **детерминированного lockstep** (как в оригинальном Generals):
по сети летят только команды игроков, симуляция идентична на всех клиентах.
Из этого следуют сквозные правила, обязательные с Фазы 0:

- **Разделение Simulation / Presentation.** Игровая логика (позиции, ХП, бой) живёт в
  детерминированном слое симуляции. Визуал (модели, анимации, эффекты, интерполяция)
  только читает состояние симуляции и не влияет на неё.
- **Фиксированный игровой тик** (например 20–30 Гц). Вся логика обновляется по тикам,
  а не по `Update`/`Time.deltaTime`.
- **Детерминизм:** никакого `UnityEngine.Random`, `Time.deltaTime`, физических флоат-эффектов
  в логике. Свой сидируемый ГПСЧ, фиксированный шаг. (Опционально — fixed-point математика;
  на старте допустимо аккуратное использование float при едином коде на всех клиентах.)
- **Команды через очередь.** Любой приказ (move, attack, build) = `Command`-объект,
  применяемый на определённом тике. Это же — основа реплеев и сетевой синхронизации.

> Если решим иначе (сервер-авторитет + репликация на Netcode for GameObjects) — Фазы 0–8
> переписываются под другую модель. Поэтому модель фиксируем до начала кодинга.

## Структура кода (asmdef-модули)

Заводим раздельные сборки сразу — это ускорит компиляцию и даст чистые границы для тестов:

```
Assets/_Project/
  Scripts/
    Sim/          (Planet.Sim)         — детерминированная симуляция, без UnityEngine-визуала
    Presentation/ (Planet.Presentation)— камера, отрисовка юнитов, эффекты, ввод-в-команды
    UI/           (Planet.UI)          — меню, HUD
    Game/         (Planet.Game)        — склейка сцен, загрузка, режимы игры
    Net/          (Planet.Net)         — транспорт, lockstep, лобби
  Tests/
    EditMode/     (Planet.Tests.EditMode)
    PlayMode/     (Planet.Tests.PlayMode)
  Prefabs/ Scenes/ Data/ Art/
```

# ФАЗЫ

## Фаза 0 — Фундамент и каркас симуляции ⭐ (бывший «пункт 3»)
Закладываем то, что потом не переписать дёшево.
- Структура папок и asmdef-модули (см. выше).
- Базовый цикл симуляции: `SimWorld` с фиксированным тиком, реестр сущностей.
- Базовый `Command` и очередь команд на тик.
- Сидируемый детерминированный ГПСЧ.
- Тонкий слой Presentation: `PresentationRoot`, который раз в кадр читает `SimWorld`
  и двигает визуальные прокси с интерполяцией между тиками.
- **Тесты:**
  - EditMode: два независимых `SimWorld` с одинаковым сидом и одинаковыми командами
    дают идентичное состояние через N тиков (тест детерминизма — главный страж).
  - EditMode: команда, поставленная на тик T, применяется ровно на тике T.

## Фаза 1 — Главное меню + сцена полигона (пункт 1)
- Сцена `MainMenu`: кнопки «Одиночная игра» (активна), «Сетевая игра», «Настройки», «Выход»
  (неактивны/заглушки).
- Сцена `Polygon` (тестовый полигон, плоский террейн + запечённый NavMesh).
- «Одиночная игра» → загрузка `Polygon`.
- Менеджер загрузки сцен.
- **Тесты:**
  - PlayMode: нажатие «Одиночная игра» загружает сцену `Polygon`.
  - PlayMode: неактивные кнопки не загружают сцен.

## Фаза 2 — Камера в стиле Generals (пункт 2)
- RTS-камера: фиксированный наклон, движение по краям экрана и WASD, зум колесом
  (в пределах min/max), опционально поворот (Q/E).
- Ограничение по границам карты.
- Камера — слой Presentation, на симуляцию не влияет.
- **Тесты:**
  - PlayMode: ввод «вправо» сдвигает камеру по +X; на границе карта камера упирается в лимит.
  - PlayMode: зум ограничен min/max.

## Фаза 3 — Данные юнитов, префабы и спавн (пункт 4)
- `UnitDef` (ScriptableObject): ХП, скорость, радиус, урон, дальность, перезарядка, фракция.
- Сущность юнита в симуляции (Sim) + визуальный префаб (Presentation) с полоской ХП.
- Спавнер: размещение стартовых юнитов двух фракций на полигоне (как в видео — пехота,
  лёгкая техника, танки).
- Привязка «sim-сущность ↔ визуальный прокси».
- **Тесты:**
  - EditMode: спавн юнита создаёт sim-сущность с ХП из `UnitDef`.
  - PlayMode: визуальный прокси следует за позицией sim-сущности.

## Фаза 4 — Выделение юнитов (пункт 5)
- Клик ЛКМ — выделить один юнит (только свои).
- Двойной клик — выделить всех однотипных в зоне видимости экрана.
- Рамка (drag) — выделить группу.
- Shift — добавить/убрать из выделения.
- Подсветка выделения + полоски ХП у выделенных.
- Контрольные группы (Ctrl+1..9 / 1..9) — опционально, но дёшево добавить здесь.
- **Тесты:**
  - EditMode: логика «точка в рамке» отбирает правильный набор сущностей.
  - EditMode: двойной клик отбирает только однотипных и только своих.
  - PlayMode: рамкой по 5 юнитам выделяются ровно 5.

## Фаза 5 — Приказы движения
- ПКМ по земле — приказ Move выделенным (через `Command`).
- Движение по NavMesh, объезд препятствий, базовое расхождение/строй группы
  (чтобы не толпились в одну точку).
- Очередь приказов по Shift (многошаговые задачи — есть в исходном диздоке).
- Индикатор точки приказа.
- **Тесты:**
  - PlayMode: после Move-команды юнит достигает цели за отведённое число тиков (±радиус).
  - EditMode: очередь из 3 точек выполняется по порядку.

## Фаза 6 — Боевая система
- Приказ Attack (ПКМ по врагу) + авто-агр в радиусе (attack-move).
- Стрельба: перезарядка, снаряд/мгновенный хит, урон, эффекты (вспышка/луч/попадание — как в видео).
- ХП, смерть, уничтожение сущности, обломки/эффект смерти.
- Таблица урон/броня (типы оружия против классов брони — из диздока) — базовая версия.
- **Тесты:**
  - EditMode: урон уменьшает ХП; при ХП≤0 сущность помечается мёртвой.
  - EditMode: перезарядка не даёт стрелять чаще интервала.
  - EditMode: тест детерминизма из Фазы 0 расширяем боем (две симуляции = идентичный исход боя).

## Фаза 7 — Базовая экономика и строительство (минимум для демо)
> Для боевого демо можно урезать, но Generals без базы — не Generals. Минимальный срез:
- 1 ресурс (деньги/припасы), 1 добывающее здание/точка дохода.
- Командный центр + 1 фабрика, очередь производства юнитов.
- Размещение здания по сетке, проверка занятости.
- **Тесты:**
  - EditMode: производство списывает ресурс и через N тиков выдаёт юнита.
  - EditMode: нельзя поставить здание на занятую клетку.

## Фаза 8 — ИИ противника (skirmish vs компьютер)
- Простой ИИ: спавн/производство, сбор армии, атака волнами, реакция на угрозу.
- Уровни сложности (хотя бы лёгкий/нормальный).
- ИИ работает в слое симуляции и детерминирован (важно для сети и реплеев).
- **Тесты:**
  - PlayMode: матч игрок vs ИИ доигрывается до условия победы без ошибок/зависаний.
  - EditMode: решения ИИ детерминированы при одинаковом сиде и состоянии.

## Фаза 9 — Сеть (lockstep): локально → LAN → онлайн
- Выбор и установка транспорта (Unity Transport / Netcode). Решение фиксируем здесь.
- Lockstep: обмен командами по тикам, согласованная задержка ввода (input delay),
  ожидание подтверждений, обработка лагов.
- Контроль рассинхрона: хеш состояния симуляции на тике, расхождение → лог/детект desync.
- Сначала 1v1, затем до 4v4.
- **Тесты:**
  - PlayMode: два инстанса в одной сессии остаются синхронны (хеши совпадают) на длинном матче.
  - PlayMode: специально подсунутый недетерминизм ловится desync-детектором (негативный тест).

## Фаза 10 — Лобби и режимы 1v1…4v4
- Создание/поиск игры, слоты команд, выбор фракции, заполнение пустых слотов ИИ.
- Старт матча с согласованным сидом и составом.
- Экран загрузки и синхронный старт.
- **Тесты:**
  - PlayMode: лобби 2v2 со смесью игрок/ИИ стартует, все клиенты входят в один сид.

## Фаза 11 — Полировка демо
- HUD: выделение, мини-карта, ресурсы, панель команд.
- Звук, эффекты, баланс первого среза юнитов.
- Условия победы/поражения, экран результатов.
- Реплей (бесплатный бонус от lockstep) — опционально.
- Профилирование производительности под массовый бой (как в видео).

### Сцены, камера и уровни (рабочий процесс)
- Объекты уровня (земля, свет, RTS-камера) — **реальные объекты сцены**, не создаются кодом.
- Собрать/обновить полигон: меню **`Planet → Setup → Build Polygon Scene`** (создаёт землю, свет, камеру, `Environment`, `GameRoot`; идемпотентно). Запустить один раз после открытия проекта.
- **Камера — единый префаб** `Assets/_Project/Prefabs/RTSCamera.prefab`, который сборщик сцены ставит на каждый уровень (правка префаба → меняется на всех картах). Создать/обновить: `Planet → Setup → Create Camera Prefab`; `Build Polygon Scene` ставит его автоматически и переводит существующую «сырую» камеру на префаб, сохраняя границы карты.
- **Параметры камеры** («ощущение»: зум, панорама, поворот, наклон) вынесены в ассет `Assets/_Project/Settings/CameraSettings.asset` — один пресет на все карты (наклон по умолчанию 60°). Префаб камеры ссылается на этот ассет.
- **Границы карты и стартовая точка** задаются на инстансе `RTSCamera` в каждой сцене (override префаба) — свои у каждого уровня. Благодаря `[ExecuteAlways]` кадр обновляется в редакторе.
- **Окружение** (деревья, камни, здания) складывать в объект `Environment` — перетаскиванием примитивов/префабов в сцене.
- **Курсор (везде):** глобальный ассет `Assets/_Project/Resources/CursorSettings.asset` применяется на старте до первой сцены (`CursorBootstrap`), поэтому курсор активен во всех сценах, включая меню. PNG лежит в `Assets/_Project/Art/Cursor/Cursor.png`; меню `Planet → Setup → Create Cursor Settings` чинит импорт текстуры (Texture Type = Cursor), создаёт ассет и подхватывает текстуру (если своего PNG нет — генерирует стрелку-заглушку). Hotspot = кончик стрелки. `CursorController` остаётся для смены курсора в рантайме (атака/нельзя).
- **Новый уровень:** `File → New Scene` → меню **`Planet → Setup → Setup Active Scene As RTS Level`** → расставить окружение → сохранить как `Assets/_Project/Scenes/<Имя>.unity`. (Скрипт сам добавит сцену в Build Settings при сборке полигона; новые карты добавляйте в Build Settings вручную или через свой пункт меню позже.)

### Как проверить
1. Открыть Unity, дождаться компиляции.
2. `Window → General → Test Runner` → EditMode → Run All (должны пройти тесты детерминизма и механик).
3. PlayMode → Run All (поток сцен; требует, чтобы обе сцены были в Build Settings — уже добавлены).
4. Открыть сцену `Assets/_Project/Scenes/MainMenu.unity`, нажать Play → «Одиночная игра» → загрузится полигон, две армии кубов сойдутся к центру (Esc — назад в меню).
